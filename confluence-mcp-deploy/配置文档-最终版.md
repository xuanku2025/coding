# Confluence / Jira MCP（mcp-atlassian）部署与运维（最终版）

目标：在一台能访问内网 Confluence/Jira 的虚拟机上运行 `mcp-atlassian`，通过内网穿透提供 **公网 HTTPS** 的 MCP Endpoint，供 **Figma Make 的 Custom Connector** 连接使用。  
本方案默认使用 **streamable-http** 传输，端点为 **`/mcp`**（不使用 `/sse`）。

---

## 1. 你将得到什么

- **一个 MCP 地址**：`https://<公网域名或 tunnel 地址>/mcp`
- **一个 Figma Connector**：同一个 Connector 内可同时调用 Confluence 与 Jira 的工具（只要在 `.env` 里都配置了）
- **虚拟机重启后自动恢复**
  - `mcp-atlassian`：通过 Docker 的 `--restart unless-stopped` 自动拉起
  - `cloudflared`：通过 systemd 服务自动拉起，并把最新地址写入文件
- **地址自动落盘**
  - `~/confluence-mcp-deploy/MCP_URL.txt`：可直接复制到 Figma 的 URL（带 `/mcp`）
  - `~/confluence-mcp-deploy/PUBLIC_URL.txt`：仅公网域名（不带路径）

---

## 2. 目录约定（便携性）

建议所有同事统一使用：

- 部署目录：`~/confluence-mcp-deploy/`
- 配置文件：`~/confluence-mcp-deploy/.env`（已在 `.gitignore` 忽略，禁止提交）

把本仓库的 `confluence-mcp-deploy/` 整个目录复制到任何同事的虚拟机即可复用流程。

---

## 2.1 一键部署（推荐，最省事）

如果你不想按步骤逐条执行，直接运行一键脚本，按提示填写一次即可自动完成：

```bash
cd ~/confluence-mcp-deploy
chmod +x bootstrap-deploy.sh
./bootstrap-deploy.sh
```

### 2.1.1 非交互部署（你本地先填好，传到虚拟机直接跑）

1) 在你本地仓库里复制模板并填写（**不要提交含密码的文件**）：

```bash
cd confluence-mcp-deploy
cp bootstrap.env.example bootstrap.env
vi bootstrap.env
chmod 600 bootstrap.env
```

2) 把整个目录传到虚拟机（示例使用 scp）：

```bash
scp -r confluence-mcp-deploy root@<虚拟机IP>:~/confluence-mcp-deploy
```

3) 在虚拟机上直接非交互执行（不会再提示你输入）：

```bash
cd ~/confluence-mcp-deploy
chmod +x bootstrap-deploy.sh
./bootstrap-deploy.sh --config ./bootstrap.env --non-interactive
```

脚本会自动：
- 生成 `.env` / `notify.env`（均不会提交到 Git）
- 启动 `mcp-atlassian`（`--restart unless-stopped`）
- 安装 `Docker`（如未安装，脚本会提示并可选自动安装）
- 安装 `cloudflared`（如未安装）
- 安装并启用 systemd：`cloudflared-mcp`
- 写入 `MCP_URL.txt` 并（若配置了 webhook）推送钉钉通知

## 3. 配置 `.env`（同时启用 Confluence + Jira）

在 `~/confluence-mcp-deploy/` 下：

```bash
cp .env.example .env
vi .env
```

### 3.1 Confluence（Server/DC 旧版本常用：用户名+密码）

```env
CONFLUENCE_URL=http://confluence.zstack.io
CONFLUENCE_USERNAME=<你的用户名>
CONFLUENCE_API_TOKEN=<你的登录密码>
CONFLUENCE_SSL_VERIFY=false
```

> 说明：`mcp-atlassian` 的故障排查文档建议：旧版 Confluence 可尝试 `CONFLUENCE_API_TOKEN` 填密码做 Basic Auth。

### 3.2 Jira（可选，填了才会启用 Jira 工具）

你说的环境是 **用户名 + 密码**，可以先按 Basic Auth 写法配置（变量名叫 token，但这里填密码）：

```env
# Basic Auth（用户名+密码）
JIRA_URL=http://jira.your-company.com
JIRA_USERNAME=<你的用户名>
JIRA_API_TOKEN=<你的登录密码>

# Server/Data Center（PAT）
# JIRA_URL=https://jira.your-company.com
# JIRA_PERSONAL_TOKEN=<pat>

# 如遇自签证书问题（可选）
# JIRA_SSL_VERIFY=false
```

> 兼容性提醒：如果你们实际是 **Jira Cloud**，通常不接受账号密码直连，需要用 API Token；此时请把密码改成 API Token，并把 `JIRA_USERNAME` 写成邮箱。

## 4. 启动 MCP（容器）

推荐命令（注意 `--restart unless-stopped`）：

```bash
docker rm -f mcp-atlassian 2>/dev/null || true

docker run -d \
  --name mcp-atlassian \
  --restart unless-stopped \
  --env-file ~/confluence-mcp-deploy/.env \
  -e MCP_VERBOSE=true \
  -p 9092:9090 \
  ghcr.io/sooperset/mcp-atlassian:latest \
  --transport streamable-http --port 9090
```

### 4.1 本机验证（关键：必须带 Accept 头）

```bash
curl -i -N -H "Accept: text/event-stream" http://localhost:9092/mcp
```

预期：HTTP 200 + `content-type: text/event-stream`。  
不带 `Accept: text/event-stream` 时可能返回 406（这是正常的协议要求）。

## 5. 内网穿透（cloudflared quick tunnel）

### 5.1 一键启动并打印 Figma URL（推荐给日常使用）

```bash
cd ~/confluence-mcp-deploy
chmod +x start-and-get-url.sh
./start-and-get-url.sh
```

脚本会：
- 确保容器在运行（不存在则创建）
- 启动 tunnel（后台）
- 输出并写入：
  - `~/confluence-mcp-deploy/MCP_URL.txt`
  - `~/confluence-mcp-deploy/PUBLIC_URL.txt`

### 5.2 配置虚拟机重启自启（推荐，运维标准做法）

```bash
cd ~/confluence-mcp-deploy
chmod +x install-autostart-systemd.sh
sudo ./install-autostart-systemd.sh

systemctl status cloudflared-mcp
cat ~/confluence-mcp-deploy/MCP_URL.txt
```

> 注意：quick tunnel 地址会变化。重启后只需要 SSH 上去 `cat MCP_URL.txt`，把新地址更新到 Figma Connector。

---

## 6. Figma Make 连接器配置

- **MCP server URL**：填 `MCP_URL.txt` 里的内容，例如：  
  `https://xxx.trycloudflare.com/mcp`

只需要创建 **一个** Connector。只要 `.env` 同时配置了 Confluence 与 Jira，Figma 通过这个 Connector 就能同时获取两个站点的信息。

---

## 6.1 （可选）地址变化自动推送到钉钉机器人

如果你不想每次重启都手动登录虚拟机 `cat MCP_URL.txt`，可以配置钉钉机器人 Webhook，让 systemd 在拿到最新地址后自动推送。

### A. 准备钉钉机器人

- 在钉钉群里添加“自定义机器人”
- 拿到 Webhook（形如 `https://oapi.dingtalk.com/robot/send?access_token=...`）
- 若开启“加签”，再拿到 `secret`（形如 `SECxxxx`）

### B. 在虚拟机上写入通知配置（不入库）

在 `~/confluence-mcp-deploy/notify.env` 写入：

```bash
DINGTALK_WEBHOOK="https://oapi.dingtalk.com/robot/send?access_token=xxx"
# 可选：机器人开启“加签”时需要
DINGTALK_SECRET="SECxxxx"
```

并确保脚本可执行：

```bash
cd ~/confluence-mcp-deploy
chmod +x notify-dingtalk.sh
```

### C. 触发方式

- **手动一键脚本触发**：执行 `./start-and-get-url.sh` 后会自动尝试发送钉钉通知
- **重启自启触发**：执行 `sudo ./install-autostart-systemd.sh` 安装的 `cloudflared-mcp` 服务，每次重启拿到新 URL 后也会自动发送

> 说明：钉钉加签依赖 `openssl`（生成签名）和 `python3`（URL encode）；若环境缺少其中之一，建议先关闭“加签”验证，或补齐依赖。

## 7. 运维与排障速查

### 7.1 常用命令

```bash
docker ps | grep mcp-atlassian
docker logs mcp-atlassian --tail 100

systemctl status cloudflared-mcp
tail -n 80 /tmp/confluence-mcp-tunnel.log

cat ~/confluence-mcp-deploy/MCP_URL.txt
```

### 7.2 常见问题

- `/sse` 404：你现在使用的是 `/mcp`，Figma URL 必须是 `.../mcp`
- `/mcp` 406：测试时没带 `Accept: text/event-stream`，按文档命令重试
- Figma 一直 Connecting：tunnel 没在跑 / 地址变了 / Figma 仍是旧 URL
- Jira 不可用：日志提示 “Jira is not configured…” → `.env` 没填 `JIRA_*` 变量或认证不对
