# 这个文件夹是干嘛用的？

这是一个**最小可复制部署包**：把内网的 **Confluence + Jira（本地部署，用户名+密码）** 通过 `mcp-atlassian` 部署成一个**公网可访问的 MCP Server**，供 **Figma Make Custom Connector** 连接使用。

它的目标是让同事拿到这个文件夹后：

- 只需要填写 **一个配置文件**：`bootstrap.env`
- 上传到虚拟机后执行 **一条命令**，就能完成部署
- 每次重启后自动恢复，并把最新地址（`https://xxx.trycloudflare.com/mcp`）**自动推送到钉钉机器人**

---

## 你会得到什么

- `MCP_URL.txt`：可直接复制到 Figma 的 MCP server URL（形如 `https://xxx.trycloudflare.com/mcp`）
- `PUBLIC_URL.txt`：仅公网域名（不带路径）
- systemd 服务：`cloudflared-mcp`（负责 tunnel 常驻/开机自启/写入最新地址/可选钉钉通知）
- Docker 容器：`mcp-atlassian`（负责连接 Confluence/Jira 并对外提供 MCP `/mcp` 端点）

---

## 适用环境

- 一台 Linux 虚拟机：能访问内网 Confluence/Jira，且能访问公网
- 有 `systemd`（支持 `systemctl`）
- Docker：没有的话脚本会提示并可选自动安装（依赖能访问 Docker 官方源）

> 备注：本包内已包含 `cloudflared-linux-amd64` 与 `mcp-atlassian_latest.tar.gz`，脚本会优先使用它们，尽量减少对外网的依赖。  
> 注意：这里自带的 `cloudflared-linux-amd64` 仅适用于 x86_64/amd64；若你的虚拟机是 arm64，需要自行准备对应架构的 cloudflared。

---

## 文件说明（你只需要关心 2 个）

- `bootstrap.env`：**唯一需要填写的配置文件**（Confluence/Jira 账号密码、端口、钉钉 webhook）
- `bootstrap-deploy.sh`：一键部署脚本（非交互模式运行）

其他脚本是部署内部用到的组件：
- `install-autostart-systemd.sh`：安装 `cloudflared-mcp` systemd 自启服务
- `notify-dingtalk.sh`：推送最新地址到钉钉群机器人
- `start-and-get-url.sh`：手动重新获取最新地址（可选）

---

## （可选）把镜像文件也一起分发（解决虚拟机拉镜像很慢）

如果虚拟机访问 `ghcr.io` 很慢，你可以在网络更好的机器上提前把镜像打成 tar，然后把 tar 文件**直接放进这个文件夹根目录**，同事部署时会自动优先 `docker load` 导入。

支持的文件名（二选一）：
- `mcp-atlassian_latest.tar`
- `mcp-atlassian_latest.tar.gz`

制作镜像包（在能拉镜像的机器上执行）：

```bash
docker pull ghcr.io/sooperset/mcp-atlassian:latest
docker save -o mcp-atlassian_latest.tar ghcr.io/sooperset/mcp-atlassian:latest
gzip -1 mcp-atlassian_latest.tar
```

## 最短使用方法

1) 编辑配置（在你本地或虚拟机都行）：
   - 从模板复制并修改 `bootstrap.env`（填 Confluence/Jira 用户名密码、端口、钉钉 webhook）

```bash
cp bootstrap.env.example bootstrap.env
chmod 600 bootstrap.env
```

2) 把整个文件夹上传到虚拟机

> **目录不强制**：可以放在任意路径（例如 `/root/confluence-mcp-deploy`）。  
> 推荐路径仍是 `~/confluence-mcp-deploy/`，方便同事按文档复制命令直接运行。

3) 在虚拟机执行（root 或有 sudo）：

```bash
cd ~/confluence-mcp-deploy   # 或 cd 到你实际放置的目录
chmod +x bootstrap-deploy.sh
./bootstrap-deploy.sh --config ./bootstrap.env --non-interactive
```

如果你希望“无论当前目录在哪”都能执行，也可以显式指定：

```bash
DEPLOY_DIR=/root/confluence-mcp-deploy ./bootstrap-deploy.sh --config /root/confluence-mcp-deploy/bootstrap.env --non-interactive
```

4) 获取可用 MCP 地址（复制到 Figma Make Connector）：

```bash
cat MCP_URL.txt
```

（可选）自检本机服务是否就绪：

```bash
curl -sS http://localhost:19092/healthz
```

## 安全注意事项

- `bootstrap.env` **包含密码/钉钉 webhook**，不要发到公共群，不要提交到 Git
- 建议上传后在虚拟机执行 `chmod 600 bootstrap.env`

## 常见问题：cloudflared 一直卡在 “Requesting new quick Tunnel...”

现象类似你在日志里看到的：

- `Requesting new quick Tunnel on trycloudflare.com...`
- 很久都没有出现 `https://xxxx.trycloudflare.com` 的公网地址

这通常不是脚本问题，而是 **虚拟机对外网/DNS 的访问受限**（例如 `trycloudflare.com` 被拦、DNS 解析失败、或只能走代理但未配置）。

你可以在虚拟机上快速判断：

```bash
getent hosts trycloudflare.com || nslookup trycloudflare.com
curl -I https://trycloudflare.com
tail -n 80 /tmp/confluence-mcp-tunnel.log
```

如果确认出网受限：

- **方案 A（推荐）**：换一台允许访问公网的虚拟机来跑 `cloudflared`
- **方案 B**：改用 `ngrok`（把 `ngrok` 二进制放到 `~/confluence-mcp-deploy/ngrok` 或 `/usr/local/bin/ngrok`，然后执行）

```bash
TUNNEL_TYPE=ngrok ./start-and-get-url.sh
```

## 运维常用命令

- **查看当前 URL**：`cat ~/confluence-mcp-deploy/MCP_URL.txt`
- **查看 tunnel 日志**：`tail -n 80 /tmp/confluence-mcp-tunnel.log`
- **重启 tunnel（会刷新公网地址）**：`systemctl restart cloudflared-mcp`
