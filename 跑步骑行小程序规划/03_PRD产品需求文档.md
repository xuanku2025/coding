# 03 产品需求文档 (PRD) - V1.0 MVP

## 1. 文档概述
本文档旨在定义“极简运动记录”微信小程序 V1.0（MVP 版本）的产品需求，为研发、设计、测试团队提供明确的功能实现标准和验收准则。

**产品理念：** 纯粹记录，即刻出发。
**核心价值：** 去除冗余社交与广告，利用微信原生能力实现稳定记录与极速分享。

---

## 2. 角色与权限说明

| 角色角色 | 描述 |
| :--- | :--- |
| **未授权用户** | 首次进入，未授权地理位置信息或未授权微信登录/昵称头像。仅能查看默认首页结构，无法开始运动。 |
| **普通跑者/骑友** | 授权位置信息（且最好为“使用小程序期间和离开后”）后，可正常记录运动轨迹、生成分享卡片、查看本地历史记录。 |

---

## 3. 核心页面信息架构与交互说明

### 3.1 首页 (Home - `/pages/index/index`)

*   **页面目的：** 快速发起一次运动。
*   **UI 布局：** 全屏地图底图为主视区。
*   **功能模块：**
    *   **顶部状态栏**
        *   当前 GPS 信号强度图标（优/良/弱）。
        *   天气简报（可选）：当前温度与天气图标。
    *   **运动类型切换 (Tab/Toggle)**
        *   **[跑步]** | **[骑行]** 按钮横排，高亮显示当前选中项。
        *   *交互：* 切换时，下方核心数据预估模块可做微小动效切换，主题色可变（如跑步使用活力橙，骑行为破风蓝）。
    *   **中心地图区**
        *   实时显示用户当前所在位置（蓝点）。
        *   *交互：* 支持拖拽查看周边、双指缩放、点击回到当前位置按钮复位。
    *   **主控组 (底部悬浮)**
        *   巨大的原型 **【开始 (Start)】** 按钮。
        *   *交互：* 点击后，若权限不足则弹窗引导；若权限正常，则出现 `3 - 2 - 1 - GO` 的全屏倒计时动画，后平滑跳转至**运动记录页**。

### 3.2 运动记录页 (In-Workout - `/pages/workout/workout`)

*   **页面目的：** 运动中稳健记录，防止误操作中断，提供清晰的数据反馈。
*   **规则限制：** 需向用户申请并配置微信小程序的 `requiredBackgroundLocation`（后台定位权限），确保切到微信聊天或手机息屏时轨迹不断。
*   **UI 布局：** 极简面板结构。
*   **功能模块：**
    *   **数据大字报 (上半区)**
        *   **主干数据：** 运动时长 (`HH:MM:SS`)，字体最大最醒目。
        *   **次要数据 1：** 运动距离 (`XX.XX km`)。
        *   **次要数据 2：** 当前配速（跑步模式，如 `5'30"/km`）或 当前速度（骑行模式，如 `25.5 km/h`）。
        *   *逻辑：* 每隔 1 秒更新一次 UI 显示；配速/速度需进行平滑算法处理，避免数值跳动过大。
    *   **轨迹追踪图 (中区/背景可视)**
        *   实时绘制用户走过的轨迹线（高亮渐变色）。
        *   *交互：* 地图自动居中并跟随用户的移动方向旋转（建议用户可选锁定北向上或车头向上）。
    *   **控制面板 (底部操作区)**
        *   **【暂停】按钮**：圆形按钮。
            *   *交互：* 点击后，进入暂停状态，计时停止，不记录轨迹点。按钮变为**【继续】**。
        *   **【结束】按钮 (安全防误触)**：
            *   *设计：* 需**长按 2 秒** 或 **滑动条解锁** 才能真正触发结束动作。
            *   *交互：* 触发后，若距离<0.1km，弹窗提示“距离过短，是否放弃保存本次记录？”；否则直接生成摘要数据，跳转至**数据总结页**，并异步上传云端归档。
        *   **【右下角：地图/面板切换】**：允许用户将上半区数据面板收起，全屏沉浸看地图。

### 3.3 数据总结与分享页 (Summary - `/pages/summary/summary`)

*   **页面目的：** 给予用户成就感，诱导其产生极简高清的社交分享行为。
*   **输入参数：** 运动 ID 或刚生成的核心摘要数据对象。
*   **UI 布局：** 基于截图的卡片流布局。
*   **功能模块：**
    *   **顶部标题与操作**
        *   返回首页按钮；“完成”按钮。
    *   **成就图表区 (Hero Section)**
        *   利用 `<map>` 组件截取包含完整轨迹线的清晰截图作为背景卡片。
        *   叠加显示本次运动的关键指标（日期时间、公里数、时长、卡路里预估、均速）。
    *   **详情数据列表**
        *   每公里配速拆解列表（如：第1公里 6'00", 第2公里 5'50"...）。
    *   **社交分享组 (核心增长点)**
        *   **【微信好友/群聊】按钮**：直接调用小程序卡片分享，卡片封面为带轨迹的小截图，标题为“我刚刚跑了 X 公里，快来挑战！”。
        *   **【生成长图海报】按钮**：
            *   *交互：* 点击进入画布海报生成界面。将静态地图、数据、自定义水印文字/贴纸通过 Canvas 绘制成一张适合发朋友圈的高清竖图。允许用户唤起系统相册选择一张照片作为背景图替换默认的地图背景。允许保存至手机相册。

### 3.4 个人中心与历史记录 (Profile - `/pages/profile/profile`)

*   **页面目的：** 数据沉淀与回顾。
*   **功能模块：**
    *   **用户概览仪表盘**
        *   拉取微信公开资料显示简单头像昵称（如未授权则显示默认头像）。
        *   核心累计数据：累计使用天数、跑步总公里、骑行总公里（数字动效滚动增加）。
    *   **按月分的历史列表**
        *   `2026年3月`
            *   `[跑步图标] 3月4日 傍晚跑 5.02km - 30:15 - 6'01"/km`
            *   `[骑行图标] 3月2日 周末骑行 20.5km - 1:10:00 - 18km/h`
        *   *交互：* 上拉加载更多。点击单条记录，可跳转至该次运动的“数据总结与分享页（查看历史态）”。
    *   **设置与关于**
        *   常亮屏幕开关。
        *   清除本地缓存/脏数据入口。

## 4. 关键异常场景与容错处理 (Exception Handling)

1.  **权限拒绝：** 若用户在开始前拒绝了位置权限，不应强行阻断整个应用，但“开始”按钮点击必须拦截并弹窗引导至系统设置页开启。
2.  **GPS 信号丢失/极弱：** (常见于进隧道、高楼地下室)
    *   **表现：** 轨迹出现飞点（如瞬间漂移了几公里）。
    *   **处理策略：** 实施粗颗粒度的**异常点过滤算法**（速度若超过博尔特极限，如每秒超 15 米的漂移点，直接丢弃该点，不参与距离累加和轨迹连线）。
3.  **微信被系统强制杀后台 (OOM)：**
    *   **表现：** 用户跑步 30 分钟后切回微信，发现小程序已经从首页重新启动，上一次记录丢失。
    *   **处理策略 (防丢包)：** 在运动过程中，每隔 N 秒（如 10 秒）将当前的经纬度数组、时间和距离通过 `wx.setStorageSync` **增量持久化存储到本地缓存**。若小程序异常崩溃重启，首页加载时检测到有未完成的“僵尸数据”，则弹窗提示`“检测到异常中断的运动记录，是否恢复？”`，用户点击恢复即可载入缓存数据继续记录。
