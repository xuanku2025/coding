# 06 部署与上线流程

当我们在开发者工具和部分真机上完成了主要功能的开发和体验优化后，项目即进入最后的打磨、测试与发布阶段。由于涉及“后台定位”等敏感权限，小程序的审核和上线需要严谨对待。

## 1. 测试验收准则 (QA Checklist)

在提交审核前，必须在内部完成以下维度的真机冒烟测试，确保核心主干不会在中途崩溃：

### 1.1 核心记录与后台保活测试
*   **[关键] iOS/Android 双端实跑：** 必须使用至少一台主流 iPhone 及其某款 Android 旗舰机型，在户外实际行走、跑步/骑行至少 2 公里以上。
*   **[关键] 切后台能力测试：** 运动开始后，将微信完全切回桌面并锁屏。等待 15 分钟后，重新打开微信进入小程序，检查这 15 分钟内的轨迹连线是否平滑完整，里程/时间数据是否正常累加，系统是否触发了 OOM 杀进程。
*   **抗干扰测试：** 运动中途接听一个长达 3 分钟的电话，再切回小程序，验证记录是否依然在后台默默进行，未丢失任何坐标点。
*   **漂移抑制测试：** 走到高楼林立处、进入隧道或地下室停留 3 分钟后再出来，检查“异常点过滤算法”是否成功剔除了乱飞的信号，没有产生荒唐的“10秒钟跑了5公里”的数据。

### 1.2 UI 与体验测试
*   **防误触测试：** 长按“结束”是否必须要按够 2 秒才生效；湿手或大量出汗时，滑动解锁是否足够顺畅但又不易误触。
*   **弱网生存测试：** 跑到没有 4G/5G 信号的山里，运动结束后点击“保存”，小程序是否能坚强地将数据先序列化存在本地缓存 `Storage` 中，等回到有网的地方再静默同步至云端。
*   **海报生成测试：** 极端长图（如跑了半马导致截图比例极大）或更换相册超大体积原图背景时，`Canvas` 绘图是否会因为内存不足而白屏崩溃；最终生成的分享图片清晰度是否适合发朋友圈，不出现明显的锯齿。

### 1.3 权限边界测试
*   首次打开彻底拒绝定位权限，界面如何优雅降级并引导。
*   在系统“设置”中将其设为“仅使用期间允许”，切后台后马上会断开连接，必须弹窗强烈建议用户修改为“使用时和离开后”。
*   未授权昵称头像时的默认展示兜底。

## 2. 账号体系与微信公众平台配置

在准备提交代码前，必须在[微信公众平台](https://mp.weixin.qq.com/)完成以下必要配置工作的查漏补缺。

### 2.1 基础信息与资质
*   **名称与 Logo：** 确认最终的极简风小程序名称（如：极简跑记、轻动轨迹等），并上传清晰的高对比度圆形 Logo。注意名称不要包含“微信”、“官方”或夸大词汇，容易被驳回。
*   **服务类目选择：** 跑步骑行类必须选择合适的类目，通常选择 **“工具 - 运动/健康”** 或 **“体育 - 体育场馆/服务”**（根据具体选项选择门槛最低的，个人开发者避免选择需要特殊资质的类目）。若涉及复杂社交（发动态评论），可能需要增值电信业务许可证，我们坚持 MVP 无内置社区，只做工具+微信原生分享，以规避此类目要求。

### 2.2 核心权限与隐私协议配置 (**极重要**)
由于我们深度依赖 `GPS 持续定位`，如果这里不配置好，代码根本无法提审，或者一提交就被秒拒。

1.  **接口权限申请 (后台定位)：**
    *   在公众平台 -> 「开发」-> 「开发管理」-> 「接口设置」中，找到并申请 **`wx.startLocationUpdateBackground`** 和相关的获取位置 API。
    *   **申请话术建议：** 附带测试阶段的录屏链接或清晰的设计图，诚恳说明：“本小程序为户外运动（跑步/骑行）记录工具，核心功能为绘制运动轨迹和计算配速，需要用户在锁屏或切后台时持续记录位置变化，否则无法计算距离，失去工具价值。现申请开通后台定位权限，仅在用户主动点击‘开始运动’且亮起提示时使用，绝不在其他场景收集位置。”
2.  **更新《小程序用户隐私保护指引》：**
    *   在公众平台 -> 「设置」-> 「服务内容声明」-> 「用户隐私保护指引」中明确声明：
        *   收集 **位置信息** 的用途：用于提供运动轨迹记录服务。
        *   收集 **相册/摄像头** 下发：用于用户自定义上传运动分享海报背景图。
        *   收集 **昵称/头像**（若需要）：用于生成带有辨识度的个人主页和分享海报。
    *   *说明：若小程序端缺少隐私弹窗的对接实现（`wx.onNeedPrivacyAuthorization`），也会被拦截无法上线。*

## 3. 部署方案 (微信云开发)

由于本项目选型了相对轻量的 **“微信云开发 (CloudBase)”**，因此部署流程大为简化，无需运维 Linux 服务器或配置 Nginx，直接在微信生态内闭环即可。

### 3.1 云环境初始化
1.  在微信开发者工具点击顶部「云开发」，创建一个新的正式环境（建议按需计费或先用极其便宜的基础个人套餐）。
2.  获取真实的环境 ID（如：`pro-record-xxx`）。
3.  在小程序前端代码 `app.js` 的 `wx.cloud.init()` 方法中，将 `env` 环境变量明确指向该正式环境 ID。

### 3.2 数据库与云存储准备
*   **集合（表）建立：** 在云开发控制台的“数据库”中，新建核心集合：`users` (存放用户偏好与统计大盘数据)、`movement_records` (存放每一次的运动摘要及其关联的 FileID)。
*   **安全规则配置：** (**重点防护**) 不能让所有人随意读写。
    *   对 `movement_records` 集合的安全规则应配置为：`"read": "doc._openid == '{openid}'"`（仅创建者可读，保障他人隐私无法遍历），`"write": "doc._openid == '{openid}'"`（仅创建者可写/改/删）。
*   **云函数上传部署：** 将 `cloudfunctions` 目录下的鉴权等逻辑代码（如处理静默登录、自动生成带参数小程序码的逻辑），右键点击 **“上传并部署：云端安装依赖 (不上传node_modules)”**。

## 4. 上线提审与发布节奏

### 4.1 提交审核
1.  **代码清理：** 移除所有的 `console.log`，关闭开发者工具中的无关调试面板，确保最终构建包去除了冗余代码，利用开发者工具的“上传代码”功能推送到云端。
2.  **提交版本：** 在公众平台「版本管理」中，将刚才上传的草稿选为“测试版”，体验无误后点击“提交审核”。
3.  **配置审核信息：** 此时最容易被拒的环节是审核员不知如何测试：
    *   务必提供一个清晰的“测试说明”。最好附上一段自己用手机拿着真正跑了一段距离、展示出轨迹画线、长按结束并生成海报过程的**演示视频（传至优酷/腾讯视频贴链接）**。
    *   否则，审核员可能坐在办公司点一下“开始”，看了一下没动静，就以“功能可用性差”为由驳回。

### 4.2 灰度发布与全量发布
一旦微信方面提示“审核通过”，我们可以采用如下节奏让产品见天日：

*   **全量发布 (针对轻量工具的首选)：** 考虑到 MVP 版本用户量从零起步，可以直接在后台点击 **“提交发布”** 使得 V1.0 面向所有人可见。
*   **线上种子用户冷启动：** 将事先准备好的一张生成极其炫酷、带有巨大“第一战”、“首杀”字样的水印海报，连同小程序卡片，发送至核心跑步群或车队群。因为小程序的“寄生”特性，只要首批使用者觉得生成的海报好看、无广告不卡顿，极易形成群内攀比晒图，这就是产品跑通的标志。
