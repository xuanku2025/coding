# 05 前端UI与功能开发计划

由于我们需要一个极简、快速、专注于核心记录和分享体验的小程序，因此在 UI 设计和前端代码架构上，也需秉持“奥卡姆剃刀”原则——如无必要，勿增实体。

---

## 1. UI 设计规范与核心页面风格

我们的目标用户可能是在户外强光下使用，或者在运动导致大汗淋漓、操作不便时使用。

### 设计准则 (Design System)

*   **极简大字号：** 核心数据（时间、距离、配速）必须采用超大字号无衬线英文字体（如 `DIN Alternate` 或 `SF Pro Display`），确保用户在快速扫视时能瞬间读懂。
*   **高对比度：** 背景以深灰/黑色为主深色模式（沉浸感更强，且 OLED 屏幕省电），配合高亮品牌色（如：**荧光绿/活力橙**用于跑步，**破风蓝**用于骑行）。
*   **大面积触控区：** “开始”、“暂停”、“结束”按钮必须足够大，且结束按钮有明确的防误触设计（需要长按 2-3 秒）。
*   **卡片式布局：** 历史记录和分享海报采用圆角卡片，提升品质感和可分享性。

### 核心页面结构参考

| 页面名称 | 视觉重心 | 关键元素层级 |
| :--- | :--- | :--- |
| **首页** | 实时地图定位 | `[定位图标]` 中心 -> `[开始大按钮]` 下方悬浮 -> `[跑步/骑行切换]` 上方 |
| **记录中** | 动态数字跃变 | `[`时间 `HH:MM:SS]` 最显眼 -> `[`距离、配速`]` -> `[`地图轨迹背景`]` -> `[长按结束/暂停 控制栏]` |
| **总结与分享页** | 带有轨迹照的海报预览 | `[`轨迹截图 + 运动数据叠加的大海报预览区`]` -> `[`保存/分享到微信/发朋友圈 操作栏`]` |
| **个人中心** | 总成就数据 | `[`用户头像昵称`]` -> `[`总里程/次数等勋章`]` -> `[`历史运动列表（按月标题分组）`]` |

## 2. 前端工程实践与目录结构

为保持代码整洁度和可维护性，我们采用以下文件结构（微信原生）：

```text
├── miniprogram/
│   ├── app.js/app.json/app.wxss      # 全局配置、样式（如统一定义的高亮色 CSS 变量）
│   ├── pages/
│   │   ├── index/                    # 首页模块 (地图展示、准备开跑)
│   │   ├── workout/                  # 记录模块 (硬核：计时、计距、防误触、后台保活逻辑)
│   │   ├── summary/                  # 结算/海报生成模块 (重点：Canvas 高清绘制逻辑、分享 API 调用)
│   │   └── profile/                  # 个人中心及历史记录列表模块 (数据请求与展示)
│   ├── components/                   # 全局复用组件库
│   │   ├── c-large-number/           # 针对运动数据定制的大数字组件（带特定字体和动画）
│   │   ├── c-slide-to-unlock/        # 滑动或长按解锁组件（结束运动防误触专用）
│   │   └── c-poster-generator/       # 封装的海报 Canvas 生成组件（核心分享引擎）
│   ├── utils/                        # 核心工具库
│   │   ├── geo.js                    # 难点：经纬度距离计算（Haversine）、异常点过滤算法
│   │   ├── format.js                 # 难点：毫秒转 HH:MM:SS、配速格式化、时间戳转日期
│   │   └── cloud.js                  # 微信云开发的基础封装（增删改查、鉴权等）
│   └── images/                       # 面包屑、图标、默认海报底图资源
└── cloudfunctions/                   # 云开发服务端函数
    └── login/                        # 处理用户鉴权，获取 OpenID 取代传统服务器 session 机制
```

## 3. 分阶段敏捷开发排期 (以 4 周前端开发计)

遵循 MVP 原则，我们先保证主干流程能跑通，再丰富视觉效果和数据持久化。

### 第 1 周：基础设施建设与跑通主干核心流
*   **目标：** 从点击开始 -> 拉取位置 -> 简单计算距离 -> 强行结束，这个最粗糙的闭环必须调通。
*   **内容：**
    *   **Day 1-2：** 搭建框架，配置微信云开发环境。完成首页基础 UI 及地图组件 `[map]` 的引入与当前位置渲染。
    *   **Day 3-5 (攻坚)：** 攻克 `wx.startLocationUpdateBackground`，跑通真机长达 30 分钟不亮屏的后台经纬度记录。这是整个立项的基础，必须首周验证其可靠性。跑不通则后续需求无从谈起。

### 第 2 周：算法优化与运动面板UI完善 (硬核计算周)
*   **目标：** 将上周拿到的一堆粗糙经纬度数据转换为用户能看懂的漂亮面板。
*   **内容：**
    *   **Day 6-7：** 编写并优化 `geo.js` 中的滤波算法。过滤掉飘到海里的点，并平滑计算总距离和瞬时配速/速度。
    *   **Day 8-9：** 开发 `workout` 记录页面的 UI。实现极具运动感的数字跳动和防误触的 `c-slide-to-unlock`（长按结束控件）。将计算好的数据实时绑定到视图层，并利用 `map.addPolyline` 随时间不断绘制运动轨迹。
    *   **Day 10：** 引入异常恢复机制：利用本地 Storage 缓存当前未结束的运动状态（处理微信崩溃或被杀后台导致的恢复逻辑）。

### 第 3 周：社交裂变与分享设计 (增长核心)
*   **目标：** 将枯燥的数据化为极有分享欲的高质量海报长图。
*   **内容：**
    *   **Day 11-12：** 开发结算页基础 UI，展示本次成绩并使用 `MapContext.takeSnapshot` 截取带轨迹的静态地图。
    *   **Day 13-14 (攻坚)：** 开发 `c-poster-generator` 组件。用 Canvas 2D API 将截取到的地图照片、用户的成绩数据大字、甚至用户相册选的照片合成在一起。排版必须考究（居中、留白、水印字体）。
    *   **Day 15：** 调用 `wx.showShareMenu` 和 `wx.saveImageToPhotosAlbum` 实现完整的分享微信群和存相册发朋友圈闭环。

### 第 4 周：数据云存储、个人中心与收尾
*   **目标：** 让工具变为能长期使用的产品，解决数据不可持续和交互粗糙的问题。
*   **内容：**
    *   **Day 16-17：** 接入微信云数据库。在运动结束后，将包含精简轨迹和数据的实体存入云端。开发 `profile` 页面，实现分页拉取历史记录列表并渲染。
    *   **Day 18：** 处理边界异常。如未授权全方位引导、GPS 信号极低提醒、没有跑出 0.1km 的抛弃弹窗。
    *   **Day 19-20：** 真机大规模封测与 UI 动画精雕细节（如页面切换的弹性阻尼、长按圆圈的填充 loading 动画等）。修复已知 bug。
