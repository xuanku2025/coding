# 04 技术框架与技术栈选型

根据小程序的定位（轻量、聚焦核心记录、重前端交互与分享），并考虑到开发效率和后续维护成本，我们对整个项目的技术栈进行如下选型和架构设计。

## 1. 整体架构概览

本项目采用 **“云开发 (Serverless) + 原生微信小程序 (或类原生跨端框架)”** 的架构。

*   **前端（客户端）：** 负责 UI 渲染、GPS 传感器数据获取、轨迹绘制、本地数据缓存、Canvas 海报生成。
*   **后端（云端）：** 采用微信云开发（WX CloudBase），提供云数据库（存历史记录）、云函数（处理分享逻辑、生成带参小程序码）、云存储（存长图或头像）。

---

## 2. 前端技术栈选型

### 方案对比

| 框架 | 优势 | 劣势 | 适用场景 | 结论 |
| :--- | :--- | :--- | :--- | :--- |
| **微信原生 (WXML/WXSS/JS)** | 性能最好直接调用底层 API 最稳定（特别是后台定位、地图绘制等硬核功能）。首屏加载极快。 | 无法跨端到 App/H5。工程化、组件化开发体验相对较弱。 | 追求极致性能、高度依赖微信独有 API 的单端项目。 | **推荐（首选）** |
| **Taro / uni-app (Vue/React)** | 使用 Vue/React 生态，开发体验好，支持多端编译。 | 会引入运行时框架，包体积略大；调用涉及底层位置和地图的复杂 API 时可能存在微小的坑或延迟。 | 团队本身熟练相关框架，未来有明确 App 化需求。 | 备选方案 |

### 最终决定：微信原生开发 + 增强工程化

鉴于运动记录应用对 **GPS 持续获取、电池功耗、后台保活、地图海量点渲染** 的要求极高，任何跨端框架的中间层都可能引入不稳定因素。因此，我们选择**微信原生开发**，但引入一些现代化的工程化手段：

*   **开发语言：** `TypeScript` / `JavaScript (ES6+)`。建议使用 TypeScript，运动数据结构复杂，强类型能减少很多 bug。
*   **CSS 预编译器：** `LESS` 或 `SASS`（微信开发者工具已内置支持）。
*   **UI 组件库：** 自定义 UI。因为我们的设计极简且定制化强（大数字面板、特殊地图控件），引入庞大的如 Vant Weapp 会增加无谓的体积，建议**纯手写核心组件**。
*   **状态管理：** 对于极简应用，使用原生的 `getApp().globalData` 或轻量级的事件发布订阅模式即可。如果页面数据联动复杂，可引入 `mobx-miniprogram`。
*   **Canvas 绘图框架：** 推荐使用 `wxml-to-canvas` 或原生的 `Canvas 2D API`。因为分享海报是核心增长点，原生 API 性能最好且不受跨端框架限制。

## 3. 后端技术栈选型

### 方案对比

| 方案 | 架构 | 优势 | 劣势 | 结论 |
| :--- | :--- | :--- | :--- | :--- |
| **自建服务器** | Node.js / Java / Go + MySQL / Redis | 彻底掌控数据，适合未来横向扩展或接入第三方硬件。 | 需要购买服务器、域名备案、配置 HTTPS、维护数据库，前期成本高。 | 暂不考虑 |
| **微信云开发 (CloudBase)** | 云数据库 (NoSQL) + 云函数 + 云存储 | **开箱即用，免鉴权 (直接获取 OpenID)**，无需购买服务器和域名，第一年几乎零成本。极其适合个人和小微团队的 MVP 版本。 | 平台绑定较深，迁移成本较高；复杂的聚合查询和高并发下可能有性能瓶颈。 | **强烈推荐** |

### 最终决定：微信云开发 (CloudBase)

*   **数据库 (Database)：** JSON 文档型数据库。用于存储用户的 Profile 和每一次的运动记录 (Record)。
*   **云存储 (Storage)：** 考虑到轨迹数据点很多（跑 10 公里可能有一两千个坐标点），为避免单个文档过大，可将**详细的轨迹坐标数组存为 JSON 文件放在云存储**，数据库中只存文件的 FileID 和运动摘要（距离、时长、平均配速）。
*   **云函数 (Cloud Functions)：**
    *   处理获取用户 OpenID 的静默登录。
    *   （可选）定时清理废弃的临时海报图片。

## 4. 核心技术难点与解决方案

在开发过程中，以下几个核心技术点将是主要挑战：

### 难点 1：微信后台持续定位与息屏保活
*   **问题：** 微信切到后台或手机锁屏后，小程序默认会被暂停执行，导致轨迹丢失。
*   **解决方案：**
    1.  **权限申请：** 必须在 `app.json` 中配置 `requiredBackgroundLocation` 权限说明。
    2.  **API 调用：** 使用 `wx.startLocationUpdateBackground` 开启后台持续定位。这不仅需要在代码中调用，还需要用户在“设置-位置信息”中选择“使用小程序期间和离开后”。
    3.  **防系统杀后台（保活）：** 结合 `wx.setKeepScreenOn` 建议用户在运动时保持屏幕常亮（如果放在袖标兜里）。此外，可利用微信内置的类似 `wx.playBackgroundAudio` 放一段无声音乐的 hack 手段（视苹果审核政策而定，谨慎使用，首选微信官方的后台定位能力）。

### 难点 2：轨迹漂移纠偏与距离计算算法
*   **问题：** 手机 GPS 在城市高楼、树林中会发生数百米的瞬间漂移，如果直接连线计算，距离会大幅偏高。
*   **解决方案 (简易滤波算法)：**
    *   **速度阈值过滤：** 在收到一个新的经纬度点时，计算与上一个点的距离除以时间差。如果算出的瞬时速度超过常理（如跑步超过 10 米/秒，骑行超过 25 米/秒），则判定为异常漂移点，直接丢弃该点不作记录。
    *   **精度过滤：** 结合定位 API 返回的 `accuracy`（水平精度），剔除精度大于 60 米的点。
    *   **距离计算：** 采用 Haversine 公式计算两个经纬度之间的球面距离。

### 3. 难点 3：Canvas 高清海报生成并带有地图截图
*   **问题：** `<map>` 组件是原生组件，层级最高且无法直接用 Canvas 绘制它。
*   **解决方案：**
    *   在结算页，当地图渲染完毕后，调用 `MapContext.takeSnapshot` 截取一张当前地图（包含轨迹）的图片。
    *   将这张截图传给 Canvas 作为底图，然后再在 Canvas 上绘制文字、半透明遮罩、用户头像等元素。
    *   最后使用 `wx.canvasToTempFilePath` 导出高清海报。
